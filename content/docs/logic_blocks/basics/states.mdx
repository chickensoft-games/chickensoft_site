**State** refers to the value maintained by a logic block.

States can receive inputs, read values from the shared blackboard, and produce outputs. Logic blocks don't actually do very much by themselves — the current state determines what inputs get handled.

States are always `record` types that derive from `LogicBlockState`, a type provided by LogicBlocks.

```csharp
public class LightSwitch : LogicBlock {
  public LightSwitch() {
    Set(new LightSwitchState.PoweredOff());
    Set(new LightSwitchState.PoweredOn());
  }
}

// Define the state. By convention, this is placed near the logic block.
// The [StateDiagram] attribute enables diagram generation.
[StateDiagram]
public abstract record LightSwitchState : LogicBlockState {
  // On state.
  public record PoweredOn : LightSwitchState, IGet<Input.Toggle> {
    public Type On(in Input.Toggle input) => To<PoweredOff>();
  }

  // Off state.
  public record PoweredOff : LightSwitchState, IGet<Input.Toggle> {
    public Type On(in Input.Toggle input) => To<PoweredOn>();
  }
}
```

Each state type can extend other state types using traditional C# inheritance, representing the idea of [compound states] from statechart theory.

State instances are pre-allocated and stored on the blackboard. Only one state is ever active at once. When a state is the active state of the logic block, we say it is `attached`. When a state change occurs, the previous state is `detached`.

### 📥 Handling Inputs & State Changes

States define input handlers which allow them to receive inputs and return the next state.

States indicate which inputs they handle by implementing `IGet<TInputType>` for each type of input they want to handle.

```csharp
public record MyState : LightSwitchState, IGet<Input.A>, IGet<Input.B>, IGet<Input.C> {
  // Don't change states on A
  public Type On(in Input.A input) => ToSelf();

  // Handle B — just stay in this state
  public Type On(in Input.B input) => ToSelf();

  // Go to StateC on C
  public Type On(in Input.C input) => To<StateC>();
}
```

Input handlers are processed synchronously and always return a `Type` indicating the next state. We use `To<TState>()` to transition to another state, or `ToSelf()` to remain in the current state.

### 🧑‍🏫 Blackboard

The **[blackboard]** is a type of dictionary collection that allows the logic block and its states to access shared data. States look up values stored in the blackboard by their system type.

The logic block keeps a blackboard internally, but doesn't expose it directly. Instead, it implements the `IBlackboard` interface, allowing you to treat it as if it were the blackboard.

```csharp
var logic = new MyLogicBlock();

// Add all the dependencies that states will need.
logic.Set<IService>(new MyRelatedService());

var service = logic.Get<IService>();
```

Each state has access to the blackboard through its built-in `Get<T>()` method.

```csharp
[Meta]
public partial class LightSwitch : LogicBlock {
  public LightSwitch() {
    Set(new MyRelatedService()); // Add to blackboard
    Set(new LightSwitchState.PoweredOn());
  }
}

public abstract record LightSwitchState : LogicBlockState {
  public record PoweredOn : LightSwitchState {
    public PoweredOn() {
      this.OnEnter(() => {
        Get<MyRelatedService>().StartDoingSomething();
      });
    }
  }
  // ...
}
```

### 🌱 Lifecycle Callbacks

#### 🚪 Entrance and Exits

States can observe when they are `entered` and `exited`. These callbacks are only invoked when the relevant part of the state's type hierarchy changes.

```csharp
public MyState() {
  this.OnEnter(() => System.Console.WriteLine("MyState entered."));
  this.OnExit(() => System.Console.WriteLine("MyState exited."));
}
```

**LogicBlocks guarantees correct behavior for entrance and exit callbacks with compound states** (states that inherit other states).

If you transition between two states that extend a common parent state, the common parent state will not have any of its exit callbacks invoked, since you are still in that _type_ of state.

<Callout type="info">
`OnEnter` and `OnExit` must be invoked as `this.OnEnter` and `this.OnExit`, respectively. LogicBlocks implements these as extension methods to properly capture the generic type context.
</Callout>

<Callout type="tip">
Use `OnEnter` and `OnExit` for subscribing and unsubscribing to events or services when entering/exiting a specific state. Because these callbacks respect the type hierarchy, they will only fire when the type of state actually changes, preventing unwanted resubscription when transitioning between sibling states.
</Callout>

### 🏎️ Stopping and Starting

By default, a logic block will not activate the first state until it is explicitly started, receives an input, or its state is read.

```csharp
var logic = new MyLogicBlock();

// Start in the state returned by GetInitialState().
logic.Start();

// Or start in a specific state type:
logic.Start<MyState>();

// Exit and detach the current state.
logic.Stop();
```

Starting and stopping a logic block triggers the relevant entrance and exit callbacks.

### ⠛ Compound / Composite States

LogicBlocks supports the concept of [compound states] (also known as composite states) as defined by statecharts. A compound state is simply a state which contains more than one state inside it.

<Callout type="info">
States can define themselves in terms of other existing states, only accounting for the changes between them. Coding-by-difference like this helps prevent duplicate logic and ensure correctness.
</Callout>

<FancyImage src="/docs/logic_blocks/basics/compound.png" alt="Logic block visual diagram" widthOverride="380px" />

To create a compound state with LogicBlocks, simply create a state record that inherits from another state type.

Below is an example of a logic block that defines a few compound states.

```csharp
[Meta]
public partial class ExerciseLogic : LogicBlock {
  public ExerciseLogic() {
    Set(new ExerciseState.Active.Walking());
    Set(new ExerciseState.Active.Running());
    Set(new ExerciseState.Inactive.Standing());
  }

  public override Type GetInitialState() => typeof(ExerciseState.Inactive.Standing);
}

[StateDiagram]
public abstract record ExerciseState : LogicBlockState {
  public abstract record Active : ExerciseState {
    public Active() {
      this.OnEnter(() => Console.WriteLine("Active"));
    }

    public record Walking : Active {
      public Walking() {
        this.OnEnter(() => Console.WriteLine("Walking"));
      }
    }

    public record Running : Active {
      public Running() {
        this.OnEnter(() => Console.WriteLine("Running"));
      }
    }
  }

  public abstract record Inactive : ExerciseState {
    public Inactive() {
      this.OnEnter(() => Console.WriteLine("Inactive"));
    }

    public record Standing : Inactive {
      public Standing() {
        this.OnEnter(() => Console.WriteLine("Standing"));
      }
    }
  }
}
```

Note how `Walking` and `Running` inherit from the same parent state, `Active`. Likewise, `Standing` inherits from the `Inactive` state.

<FancyImage src="/docs/logic_blocks/basics/call_order.png" alt="Logic block visual diagram" widthOverride="380px" />

States that inherit from other states will register their enter/exit callbacks in the order that the C# constructors are executed: from base type to most derived type.

When the example state machine shown above is started with `Standing`, the entrance callbacks registered by `Inactive` and `Standing` will both be invoked.

```csharp
logic.Start<ExerciseState.Inactive.Standing>();

// Prints:
// Inactive
// Standing
```

If the `Standing` state transitions to `Walking`, all the entrance callbacks for `Walking` would execute:

```text
Active
Walking
```

Transitioning from `Walking` to `Running`, however, will result in only the following being printed:

```text
Running
```

Because the `Running` state also inherits from the `Active` state, transitioning between one `Active` state (`Walking`) to another `Active` state (`Running`) only results in the entrance callbacks being invoked for the portion of the state hierarchy that has changed — i.e., only the callbacks for each subtype of the last common ancestor type.

<Callout type="note">
LogicBlocks determines which entrance/exit callbacks to run when changing states, and only calls the entrance or exit callbacks that actually apply to the state transition. This behavior is consistent with the statechart requirements for [compound states]:

> When a substate transitions to another substate, both “inside” the compound state, the compound state does not exit or enter; it remains active.
</Callout>

### ⚙️ Adding Inputs from Callbacks

Entrance and exit callbacks (and event subscriptions) are great places to add inputs that force state changes.

```csharp
public record MyState : ExerciseState, IGet<Input.SomethingHappened> {
  public MyState() {
    this.OnEnter(() => Get<ISomeService>().DataReceived += OnDataReceived);
    this.OnExit(() => Get<ISomeService>().DataReceived -= OnDataReceived);
  }

  private void OnDataReceived(int data) {
    // Trigger an input on the logic block that owns us.
    Input(new Input.SomethingHappened(data));
  }

  public Type On(in Input.SomethingHappened input) => To<OtherState>();
}
```

We subscribe to a service when entering the state and unsubscribe when exiting. When the service fires an event, we add an input to the logic block — effectively forcing a state change reactively.
