Logic block states can easily be unit tested by themselves. To understand how to test a state, you need to know a little bit about its underlying context.

## 🕵️ Understanding the Context

Each logic block state has an internal `context` which wraps the logic block it is attached to.

Behind the scenes, calling the state's `Input`, `Output`, and `Get` methods require the state to be the *active* state. Attempting to call those state methods when the state is not the currently active state on a logic block results in the state throwing an exception. This most commonly happens if you reference one of those methods in the state's constructor instead of in the relevant lifecycle callbacks, since the state is not active in a logic block at the time it is constructed.

## 🥸 Using the State Tester

When testing states, call `state.Test()` to get a `StateTester` — a utility with its own blackboard that lets you collect inputs and outputs produced by the state. State testers make it trivial to verify that a state behaves as expected.

```csharp
[Fact]
public void PlaysBeepingSoundOnEnter() {
  var state = new TimerState.PoweredOn.Beeping();

  // Create a state tester (fake context) for testing purposes.
  var tester = state.Test();

  // Simulate the state being entered.
  state.Enter();

  // Verify that the state produced the outputs we expect.
  tester.Outputs.ShouldBe([new Output.PlayBeepingSound()]);
}
```

You can also verify inputs added during testing in the same way via `tester.Inputs`.

Here's another example where we add a dependency to the tester's blackboard. This allows us to swap the dependencies a state uses during testing.

```csharp
[Fact]
public void ChangesDuration() {
  var state = new TimerState.PoweredOn.Idle();

  var tester = state.Test();

  // Put a value on the blackboard for the state to use.
  tester.Set(new TimerState.Data() { Duration = 30.0d });

  var duration = 45;

  // Call the input handler directly and check it returns the expected state type.
  var nextState = state.On(new TimerInput.ChangeDuration(duration));
  nextState.ShouldBe(typeof(TimerState.PoweredOn.Idle));

  tester.Get<TimerState.Data>().Duration.ShouldBe(duration);
}
```

<Callout type="note">
Input handler methods return `Type` in LogicBlocks 6. To verify the next state, compare against the expected `typeof(...)` value.
</Callout>

## 🚪 Testing Entrance and Exit

Entrance and exit callbacks can be invoked manually by calling `Enter()` and `Exit()` on the state.

```csharp
var tester = state.Test();

// Simulate the state being entered.
state.Enter();

// Simulate the state being exited.
state.Exit();
```

You can pass a `LogicBlockState` instance as the `previous` (for `Enter`) or `next` (for `Exit`) argument to control which callbacks in the type hierarchy are invoked. This is useful for testing a derived state in isolation without triggering base class callbacks.

```csharp
var state = new TimerState.PoweredOn.Idle();
var previousState = new TimerState.PoweredOn.Beeping();
var tester = state.Test();

// Passing a state that derives from PoweredOn as the "previous" state tells LogicBlocks
// that we're already in a PoweredOn context — so only the Idle-specific
// entrance callbacks (if any) will run, not the PoweredOn ones.
state.Enter(previousState);
```

<Callout type="info">
Passing a previous/next state is optional and usually only done when testing a state that derives from another state you have separate unit tests for.
</Callout>

## 🔍 Resetting and Clearing the Tester

The `StateTester` can be reset or cleared between test cases:

```csharp
var state = new TimerState.PoweredOn.Beeping();
var tester = state.Test();

state.Enter();

// Reset clears inputs, outputs, and history but keeps blackboard data.
tester.Reset();

// Clear resets everything including the blackboard.
tester.Clear();
```
